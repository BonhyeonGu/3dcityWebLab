2. 3DCityDB Software Suite 
관계형 데이터베이스 스키마(Relational Database Schema)는 3D City Database(3DCityDB)의 핵심 구성 
요소이다. 매핑 규칙은 CityGML의 객체 지향 데이터 모델을 관계형 데이터베이스 스키마에 수동으로 매
핑하는 데 사용되었다. 데이터베이스 스키마는 공간 데이터 유형을 사용하여 CityGML 개체의 기하학적 
속성을 나타낸다. 현재 3DCityDB는 두 가지 다른 공간 관계형 데이터베이스 관리 시스템(SRDBMS)을 지
원한다. 첫 번째는 상용 SRDBMS ORACLE Spatial/Locator이고 두 번째는 PostGIS 확장이 포함된 오픈 
소스 SRDBMS PostgreSQL이다.
결과적인 데이터베이스 스키마는 완전히 자동으로 데이터베이스 스키마를 생성하는 시스템에 비해 
3DCityDB가 더 적은 수의 테이블을 요구한다는 점에서 더 콤팩트하다. 수동 매핑은 또한 테이블 및 속
성 이름이 CityGML 데이터 모델의 각 클래스 및 속성 이름과 동일하거나 적어도 유사하도록 한다. 이렇
게 하면 데이터베이스의 테이블과 직접 상호 작용해야 하는 사용자가 쉽게 작업할 수 있다. 크고 복잡
한 도시.GML 데이터셋은 데이터베이스 언어 SQL을 사용하여 중앙 데이터 저장소 내에서 효율적으로 관
리, 분석 및 쿼리할 수 있을 뿐만 아니라 외부 GIS 및 ETL 소프트웨어 응용 프로그램에서 쉽게 액세스
하여 예를 들어 해당 데이터베이스 테이블에 정보를 추가하여 3D 도시 모델을 풍부하게 할 수 있다.
데이터베이스 스키마 외에도 3D 오브젝트 및 전체 3D 도시 모델의 경계 볼륨 계산, 객체 삭제 또는 공
간 인덱스 관리와 같은 기본 작업을 수행하기 위해 저장 프로시저가 각각 PL/SQL(ORACLE) 또는 
PL/pgSQL로 제공된다. 또한 3D CityDB에는 수많은 소프트웨어 도구가 포함되어 있다(cf). 그림 1), 주제
(thematic) 및 공간 필터 기준에 따른 3D 도시 모델 데이터를 KML, COLLADA, glTF와 같은 3D 시각화 
형식과 스프레드시트 형식으로 유연하게 추출할 수 있다.

1.1. 시스템 요구사항
1.1.1 3D City Database
3D City Database의 인스턴스를 설정하려면 PostgreSQL 또는 Oracle database 설치가 필요하다. 
PostgreSQL (https://www.postgresql.org/)
지원 버전은 PostGIS Extension 2.3 이상 가지는 PostgreSQL 9.6 이상이다. 또한 항상 최신 패치와 
업데이트를 설치한다. (2022년9월 기준 PostgreSQL 버전 14.5)
Oracle (https://www.oracle.com/database/)
지원 버전은 Oracle 10g R2 이상입니다. 3D City Database를 사용하려면 Oracle Spatial 또는 
Locator 확장을 통해 제공되는 공간 데이터 지원이 필요하다. 예기치 않은 오류를 방지하고 최신 기
능을 활용하려면 사용 가능한 패치를 설치하는 것이 좋다. Oracle 10g R2의 경우 
KML/COLLADA/glTF 내보내기 기능을 사용하려면 최소 패치 세트 10.2.0.4.0 이상이 필요다.
3D CityDB 스키마를 생성하기 위한 SQL 스크립트는 데이터베이스 시스템의 기본 명령줄 클라이언트(즉, 
PostgreSQL에 psql, Oracle에 SQL*Plus)에 의해 실행되도록 작성된다. 스크립트에는 이러한 클라이언트
에 특정한 메타 명령이 포함되어 있으며, 다른 클라이언트 소프트웨어를 사용할 경우 제대로 작동하지 
않는다. 따라서 3D City Database를 설정하려는 기기에 psql 또는 SQL*Plus가 설치되어 있는지 확인해
야 한다.
1.1.2. Importer/Exporter Tool
Importer/Exporter Tool은 Java 8 이상을 지원하는 모든 플랫폼에서 실행할 수 있다. Importer/Exporter

실행을 위한 권장 버전은 Java 11이다. 이 tool은 다음 운영 체제에서 성공적으로 테스트되었다 (단, 이
에 국한되지는 않음).
Microsoft Windows 7, 8, 10, 11;
Apple Mac OS X 및 Mac OS;
Ubuntu Linux 9 ~ 22.
Importer/Exporter Tool을 설정하기 전에 시스템에 Java Runtime Environment(JRE)가 설치되어 있어야 
한다. Java 설치 패키지는 여러 공급업체에서 제공하고 관리한다. 다음 목록은 무료로 다운로드하여 사
용할 수 있는 Java 배포판의 전체 개요를 제공한다.
Eclipse Temurin (https://adoptium.net/)
Amazon Coretto (https://aws.amazon.com/ko/corretto/)
OpenJDK (https://openjdk.org/)
Oracle Java 17 LTS (https://www.oracle.com/java/technologies/downloads/)
운영 체제의 설치 지침을 따르도록 한다. Java 17 LTS(Long-Term Support) 버전부터 Oracle Java는 다
시 무료 사용 라이센스로 배포되고 이전 버전의 Oracle Java 사용은 유료 구독 라이센스로 제한된다. 마
찬가지로, Azul Zulu나 Red Hat OpenJDK와 같은 다른 Vendor들의 자바 바이너리도 상업적 이용이나 
업데이트에 대한 접근이 필요할 수 있다. 공급업체에서 제공하는 사용 약관을 주의 깊게 검토해야 한다.
Importer/Exporter에는 설치 프로세스 단계를 안내하는 설치 프로그램이 함께 제공된다. 데이터베이스 
스크립트, 플러그인 및 테스트 데이터셋을 포함하여 Importer/Exporter를 완전히 설치하려면 약 280MB
의 하드 디스크 공간이 필요하다. 필수 응용 프로그램 파일만 설치하면 약 160MB의 하드 디스크 공간
이 사용된다. 설치 패키지는 설치 프로세스 중에 선택할 수 있다.
Importer/Exporter는 기본적으로 1GB의 메인 메모리로 실행된다. 사용 가능한 최대 기본 메모리는 JVM 
기본 옵션에 의해 제어된다. 이 설정은 대부분 플랫폼과 대부분의 Importer/Exporter 프로세스에서 적절
해야 한다. 필요한 경우 프로그램의 시작 스크립트에서 또는 환경 변수를 사용하여 메인 메모리 제한을 
수동으로 조정할 수 있다. (자세한 내용은 Section 4.1을 참조하십시오.)

1.2. Importer/Exporter 설치
GitHub 릴리스 섹션(https://github.com/3dcitydb/3dcitydb-suite/releases) 또는 3D City Database 웹 사
이트(https://www.3dcitydb.org)에서 설치 프로그램 3DCityDB-Import-Exporter-{version}-Setup.jar를 다운

정 및 데이터베이스 스크립트, 3D Web Map Client, Importer/Exporter용 옵션 플러그인 및 테스트 데이
터셋도 번들로 제공한다.
[그림] Github 릴리즈 섹션 (https://github.com/3dcitydb/3dcitydb-suite/releases)에서 제공되는 3D City

1.2.2. Command-line 설치
설치 Wizard는 Headless 모드, 즉 GUI 없이 실행할 수 있다. 예를 들어, 대상 컴퓨터가 GUI를 제공하지 
않거나 SSH 또는 유사한 수단을 통해 원격 세션에서 설치가 수행되는 경우에 유용하다. GUI 모드가 아
닌 콘솔 모드에서 설치 프로그램을 실행하려면, 아래 표시된 -console 옵션을 사용하면 된다.
$ java -jar 3DCityDB-Importer-Exporter-{version}-Setup.jar –console
GUI 모드와 유사하게 설치 Wizard는 설치 프로세스의 단계를 안내하며, 설치를 완료하려면 각 단계에서 
사용자 입력이 필요하다. 예를 들어 GUI 모드와 같은 선택적 소프트웨어 패키지 중에서 선택할 수도 있
습니다.

1.3.1. Shell Scripts

사용할 수도 있다.

1.3.3. PostgreSQL 설치 단계 
Step 1 - 빈 PostgreSQL 데이터베이스 만들기
슈퍼유저 또는 CREATEDB 권한이 있는 사용자를 선택하여 PostgreSQL 서버(예: 'citydb_v4')에 새 데이터
베이스를 생성한다. 3D City Database 스키마를 설정할 새 데이터베이스의 소유자로 사용자를 선택하거
나 생성한다. 단계에서 이 사용자를 'citydb_user'라고 한다. 데이터베이스 소유자가 아닌 사용자로 스키
마를 설정하려면 이 사용자에게 적절한 권한을 부여해야 한다.
데이터베이스에 연결하고 입력하십시오.
CREATE DATABASE citydb_v4 OWNER citydb_user;

CREATE_DB 스크립트를 실행한 후 3D City Database에서 사용할 좌표 참조 시스템(CRS)을 입력하라는 
메시지가 표시된다. 대부분의 경우 CRS의 EPSG 코드와 유사한 CRS의 PostGIS 특정 SRID(Spatial 
Reference ID)을 입력해야 한다. 공간 참조(Spatial Reference)를 정의하기 위한 3개의 프롬프트가 있다.
(1)
첫째 데이터베이스의 지오메트리 열에 사용할 SRID를 지정한다. 이전 버전의 3D City Database와 
달리 기본 CRS가 정의되어 있지 않다.
(2)
둘째, 데이터에 실제 3D CRS가 사용되지 않는 경우 높이 시스템의 SRID를 지정한다. 이것은 메타
데이터로 간주될 수 있으며 데이터베이스의 지오메트리 열에는 영향을 미치지 않는다. 기본값은 
0이고, "설정되지 않음"을 의미한다.
(3)
셋째, CRS의 GML 호환 URN(Uniform Resource Name) 인코딩을 제공한다. 기본값은 OGC 네임스
페이스를 사용하며 처음 두 개의 사용자 입력으로 구성된다: 
urn:ogc:def:crs,crs:EPSG::<crs1>[,crs:EPSG::<crs2>]
SRID 및 URN 인코딩에 대한 자세한 내용은 3.3절에서 확인할 수 있다.

아니다(또는 해당 데이터베이스에 개체를 만들 수 있는 권한이 없다).;
●
PostGIS 확장이 설치되지 않았다.
●

Step 5 – 설정 점검
3D City Database는 citydb라는 별도의 PostgreSQL 스키마에 저장된다. 저장 프로시저는 citydb_pkg라
는 별도의 PostgreSQL 스키마에 기록된다. 일반적으로 모든 쿼리에서 dot 표기법을 통해 서로 다른 스
키마를 다루어야 한다.
SELECT * FROM citydb.building;
다행히 해당 스키마가 데이터베이스 검색 경로에 있을 때는 이러한 문제를 피할 수 있다. 검색 경로는 
설정 중에 자동으로 조정됩니다. 검색 경로에 citydb, citydb_pkg, 및 public 스키마가 포함되어 있는지 
확인하기 위하여 (PostGIS 요소에 필요하다) 다음 명령을 실행한다.
SHOW search_path;
※ Note
생성된 3D City Database를 새 데이터베이스의 템플릿 데이터베이스로 사용할 때, 검색 경로 정보가 
전송되지 않으므로 새 데이터베이스에 대해 다시 설정해야 한다. 
ALTER DATABASE new_citydb_v4 SET search_path TO citydb, citydb_pkg, public;
검색 경로는 동일한 세션 내에서가 아니라 다음 로그인 시 업데이트될 것이다.
모든 데이터가 포함된 3D City Database를 삭제하려면 CREATE_DB와 동일한 방법으로 DROP_DB 스크
립트를 실행한다. 단순히 'citydb'와 'citydb_pkg' 스키마를 연속적으로 삭제하는 것만이라도 작업이 완료
된다.

1.3.4 오라클의 설치 단계
Step 1 - 3D City Database의 사용자 정의
3D City Database에서 작업할 전용 데이터베이스 사용자를 만들어야 한다. 이 사용자는 CONNECT 및 
RESOURCE 역할이 할당되어 있어야 하며, CREATE SEQUENCE 및 CREATE TABLE 권한이 있어야 한다. 
※ Note
공간 인덱스를 활성화 및 비활성화하려면 CREATE SEQUENCE 및 CREATE TABLE 권한이 필요하다. 
역할을 통해 이러한 권한을 상속하는 것만으로는 충분하지 않다.
Step 2 – CONNECTION_DETAILS[.sh | .bat] 스크립트 편집
3dcitydb/oracle/ShellScripts 디렉토리로 이동하여 운영 체제에 해당하는 폴더를 선택하고 텍스트 편집기 
내에서 CONNECTION_DETAILs라는 파일을 연다. DBMS에 연결하는 데 사용할 변수는 5개이다. 
SQL*Plus가 시스템 경로에 이미 등록되어 있으면 SQLPLUSBIN 변수의 디렉터리를 설정할 필요가 없다. 
다른 매개 변수는 Oracle 사용자에게 분명해야 한다. 
다음은 Windows에서 전체 CONNECTION_DETAIL을 표시하는 예입니다.
set SQLPLUSBIN= C:\\Oracle\\instantclient_11_2 ::Directory containing the SQL*Plus binary
set HOST=localhost                             ::Name of the database server
set PORT=1521                                  ::Port of the database server
set SID=orcl                                   ::SID of the 3DCityDB database to connect to
set USERNAME=citydb_user                         ::Database user to connect with
※ Note
읽기 액세스 권한을 부여하거나 취소하는 스크립트에는 SYSDBA 권한이 필요하다. 
SYSDBA_USERNAME이라는 추가 매개 변수를 사용하여 CONNECTION_DETAILES 스크립트에서 
SYSDBA 사용자를 지정할 수 있다.
Step 3 - CREATE_DB 스크립트 실행
데이터베이스 자격 증명(database credentials)이 정의되는 즉시 CREATE_DB 스크립트를 실행한다. 
CONNECTION_DETAILS와 동일한 폴더에 있다(1.3.1절 참조).
Step 4 - 좌표 기준 시스템(CRS; Coordinate Reference System) 정의
PostgreSQL와 같이 사용자는 지오메트리 열에 사용되는 SRID, 높이 시스템의 SRID 및 CRS의 GML 호환 
URN 인코딩을 입력하라는 메시지가 표시된다(자세한 내용은 3.3절 참조).
Step 5 – 버전 관리(Versioning) 활성화 또는 비활성화
CRS 정보를 제공한 후, 사용자는 데이터베이스의 버전을 사용할지 여부를 묻는다. 버전 관리는 Oracle의 
Workspace Manager 기능을 기반으로 구현된다 (자세한 내용은 Oracle 설명서 참조). '예' 또는 '아니오'
를 입력하고, Enter 키를 누르면 기본값 'no'가 확인된다. 일반적으로 버전 지원 테이블의 삽입, 업데이
트, 삭제 및 인덱스 작업은 버전 지원이 없는 테이블에 비해 상당히 많은 시간이 소요된다.

Step 6 – Spatial 또는 Locator 라이센스 옵션 선택
Spatial 또는 Locator 지원을 통해 Oracle 데이터베이스에 3D City Database 인스턴스를 설정할 수 있다. 
Locator는 사용 가능한 공간 데이터 유형과 관련하여 Spatial과 다르므로 Oracle 설치에 유효한 라이센
스 옵션을 지정해야 한다. Locator의 경우 'L'을, Spatial(기본값)의 경우 'S'로 선택할 수 있다.
※ Note
Locator에는 GeoRaster 데이터 유형이 없기 때문에 Locator를 선택할 때 Raster 
Reliefs(RASTER_RELIFF, GRID_COVERage, GRID_COVERage_RDT)를 저장하는 3D City Database 테이
블이 생성되지 않는다.
Oracle Spatial에서 사용할 수 있는 여러 공간 작업과 기능은 Oracle 설치에서 사용할 수 있더라도 
Locator 라이센스의 적용을 받지 않는다. 데이터베이스 사용자는 Oracle 라이센스 옵션을 준수해야 
한다. 3D City Database를 설정할 때 Locator 또는 Spatial을 선택해도 라이센스 옵션이나 사용자의 
책임에는 영향을 주지 않는다.
Step 7 - 설정 점검
설정 절차가 완료되면, 테이블, 시퀀스, 및 패키지(저장된 프로시저가 포함된)가 사용자 스키마에 나타난
다. 
데이터베이스 버전 관리는 ENABLE_VERSIONING과 DISABLE_VERSIONING 스크립트에 의해 언제든지 켜
거나 끌 수 있다. 이 스크립트들은 Oracle Workspace Manager의 루틴을 호출하고, 3D City Database 
인스턴스에 저장된 데이터 양에 따라 실행하는 데 시간이 걸린다.
마지막으로, 3D City Database의 스키마 및 저장 프로시저는 CREATE_DB와 같이 실행되는 DROP_DB 스
크립트를 사용하여 삭제할 수 있다. CREATE_DB와 마찬가지로 라이센스 옵션(Locator 또는 Spatial)을 제
공해야 한다. 스크립트는 3D City Database 스키마에 저장된 모든 데이터를 삭제할 수 있으나, 데이터베
이스 사용자는 삭제되지 않는다.

1.4. 이전 릴리스로부터의 Migration
기존 3D CityDB 인스턴스를 최신 버전으로 Migration하는 셸 스크립트는 Importer/Exporter 도구의 설
치 디렉터리 내 ShellScripts/[Windows|Unix]/Migration 폴더에 있다.
예를 들어 3D CityDB 인스턴스를 버전 2.x 또는 3.x에서 버전 4.x로 업그레이드하려는 경우, Major 
version 업데이트를 위한 Migration 스크립트가 필요하다. minor 버전 업데이트(예: 버전 4.x에서 4.y로)
의 경우 별도의 업그레이드 스크립트가 제공된다(1.4.4절 참조). Migration에는 3D CityDB 테이블 또는 
개체가 변경될 수 있으므로 Migration을 시작하기 전에 데이터를 백업해야 한다.
※ Hint
또 다른 안전하고 간단한 Migration 방법은 이전 버전의 Importer/Exporter와 함께 v2.x/v3.x 인스턴
스의 데이터베이스 내용을 CityGML로 내보내고 이 릴리스와 함께 제공된 새 Importer/Exporter를 사
용하여 데이터를 새 3D City Database 버전으로 다시 가져오는 것이다. 그러나 데이터베이스에 저장
된 데이터 양에 따라 이 접근 방식은 시간이 더 소요될 수 있다.
※ Note
Migration 스크립트는 Oracle에서 버전-활성화된 테이블을 처리하지 않는다. 따라서 Oracle을 사용 
중이고 버전 관리를 활성화한 경우 데이터를 내보내고 다시 가져오는 것이 새 3D CityDB 버전으로 
Migration 하는 데 권장되는 방법이다.
Migration 프로세스를 시작하려면 MEGRIDGE_DB 셸 스크립트를 실행하시오. CONNECTION_DETAILS 파

1.4.1. PostgreSQL에서 v2에서 v4로 Migration
Step 1 – MIGRATE_DB 실행
PostgreSQL에서 3DCityDB의 기존 v2.x를 Migration 하려면 MRISTARY_DB 셸 스크립트를 실행하고 값 2
를 첫 번째 사용자 입력으로 선택하면 된다.
Step 2 – 고유한 텍스처 URI를 사용해야 함
3D City Database의 v3부터 둘 이상의 지오메트리에서 참조되는 텍스처는 더 이상 SURFACE_DATA 테
이블에 중복으로 저장되지 않고 TEX_IMAGE 테이블에 한 번만 저장된다. 텍스처 URI가 고유하고 다른 
텍스처 파일에 사용되지 않는다는 것을 보장할 수 있는 경우 Migration 프로세스 중에 이 최적화를 적
용할 수도 있다. 그렇지 않으면 Migration 중에 일부 텍스처가 손실되고 나머지 이미지가 잘못된 표면을 
참조하게 된다. 
데이터베이스에 중복된 텍스처 URI가 없는 경우 Migration 스크립트의 두 번째 입력으로 'y' 또는 'yes'
를 선택하여 최적화하도록 한다. 데이터베이스의 텍스처가 동일하게 지정된 경우(또는 모르는 경우) 'n'
(기본값이기도 함)을 입력하여 스크립트를 실행할 수 있다. v2의 SURFACE_DATA 테이블의 TEX_IMAGE 
컬럼의 항목들은 v4의 TEX_IMAGE 테이블에 1:1로 맵핑된다.
※ Note
URI가 다른 동일한 텍스처는 다른 이미지 파일로 간주되어 새로운 TEX_IMAGE 테이블에 모두 저장
되기 때문에 Migration 전에 텍스처 URI의 단순한 통합은 텍스처를 한 번만 저장하는 데 도움이 되
지 않는다. 이진 데이터 자체를 비교해야 한다.
Step 3 – 설정 점검
v2.x 스키마의 데이터 선택, 참조 업데이트 및 데이터베이스 개체 생성을 보고하는 일련의 로그 메시지
가 출력되면 스크립트는 '3DCityDB Migration Complete!' 메시지로 완료된다. 이전 데이터베이스 스키마
가 Migration 중에 삭제되지 않은 경우(마지막 단계 참조) 두 3D City Database 버전은 모두 하나의 데
이터베이스에 유지된다. 모든 것이 올바르게 전송되었는지 더 비교할 수 있기 때문에 사실 좋은 것이다.

1.4.2 Oracle에서 v2에서 v4로 Migration
Step 1 – 기존 설치 업그레이드
v4.x로의 Migration은 3D City Database의 버전 2.1.0 인스턴스에서 수행해야 한다. 2.1.0 이전 버전은 
도시 객체의 내부 저장 공간이 크게 변경되었기 때문에 2.1.0으로 먼저 업그레이드해야 한다. 해당 업그
레이드 스크립트는 v2.1.0 릴리스와 함께 제공된다. 2.1.0으로의 업그레이드는 모든 이전 버전 2.0.0에서 
2.0.6으로 수행할 수 있다. 업그레이드 절차에 대한 자세한 설명은 버전 2.1.0 설명서를 참조하시오.
3D City Database를 업그레이드하기 전에 데이터베이스 백업을 통해 모든 데이터를 보호하는 것이 좋
다. 후자는 Importer/Exporter 도구를 사용하거나 Oracle에서 제공하는 도구를 사용하여 쉽게 수행할 수 
있다.
※ Note
업그레이드 프로세스의 마지막 단계는 시간이 오래 소요된다. 대규모 및/또는 버전이 지정된 데이터베
이스에 있는 모든 도시 객체의 내부 저장소를 변경하는 데 몇 시간이 소요 될 수 있다. 초기 상태에 
따라 공간 인덱스는 프로세스에서 비활성화되었다가 다시 활성화되어 전체 기간을 추가될 수 있다. 
이 프로세스는 일관성이 없는 상태를 이어질 수 있으므로 중단되어서는 안 된다. 데이터베이스 업그
레이드를 시작하기 전에 모든 데이터를 백업하는 것이 일반적으로 권장되는 방법임을 참고하시오.
Step 2 – 새 설치 생성
Migration 스크립트는 v2.1.0이 설치된 사용자 스키마에서 3D City Database 스키마 v4를 포함해야 하
는 다른 사용자 스키마로 데이터를 전송한다. 아직 설치되지 않은 경우 1.3절에 설명한 데로 새 버전을 
설치한다.
Step 3 – v2.1.0 스키마에 대한 선택 권한을 v4 스키마에 부여
Migration 프로세스를 수행하려면 v4 스키마를 사용하는 사용자가 v2.1.0 버전을 사용하는 사용자 스키
마에 액세스할 수 있어야 한다. 따라서 GRANT_ACCESS_V2 셸 스크립트(1.3.1절 참조)를 v2 사용자로 실
행한다. 실행 시 사용자는 3D City Database v4 인스턴스의 스키마 이름을 입력해야 한다.

Step 4 – MIGRATE_DB 실행
이제 GRANT_ACCESS_V2와 같은 폴더에 있는 MRISAGE_DB 스크립트를 v4 사용자와 시작한다. 값 2를 
첫 번째 입력으로 선택하고 v2.1.0 인스턴스가 있는 스키마 이름을 지정한다.
Step 5 – 고유한 텍스처 URI를 사용해야 함
PostgreSQL 버전과 같이 다른 이미지에 사용되는 중복 텍스처 URI가 없는지 확인하거나 'n'을 입력하여

하시오.
Step 6 – Spatial 또는 Locator 라이센스 옵션 선택
마지막 입력 매개 변수를 사용하여 3D City Database의 v4 인스턴스를 설정할 때처럼 Oracle 서버에서 
실행되는 데이터베이스 라이센스를 지정한다. Spatial(Raster data를 추가로 migrate함)의 경우 'S'를 선택
하고 Locator의 경우 'L'을 선택한다.
Step 7 – 설정 점검
스크립트 임시로 데이터베이스 인덱스와 외부 키 제약 조건을 비활성화하고 Migration 절차
(CITYDB_MIGRATE)가 있는 추가 패키지를 생성한다. Migration 진행이 완료되면 패키지가 다시 제거되고 
콘솔에 "DB migration is completed." 메시지가 출력된다. 새 사용자 스키마의 데이터베이스 보고서를 
생성하고 3D City Database의 2.1 인스턴스를 포함하는 스키마 보고서(이전 버전의 Importer/Exporter 
도구로 수행)와 비교하는 것이 좋다. 다음 사항을 확인한다.
●
누락된 도시 개체가 없다(데이터베이스 report 수행).
●
인덱스와 외부 키가 다시 활성화되었다.
●
특징과 속성 사이의 관계가 올바르고
●
뷰어 응용 프로그램에서 exports가 올바르게 보인다.
Step 8단계 – Deprecated v2.x 스키마 삭제
Migration이 완료된 경우 v2.x 사용자는 DROP_DB(버전 2.x)를 호출하여 더 이상 사용되지 않는 스키마
를 삭제하기만 하면 된다. v2.x 사용자 삭제도 작동한다.
1.4.3. v3에서 v4로의 Migration
v3에서 v4로의 Migration 프로세스에서는 MIGRATE_DB 스크립트에 '3' 값을 입력한 후 사용자 입력이 
필요하지 않습니다(Oracle에서 라이센스 선택 제외).
※ Note
기존 테이블에 대한 스키마 변경사항은 ALTER TABLE 문과 함께 적용되며, ALTER TABLE 문에 수백
만 개의 행이 포함되어 있으면 해당 테이블을 더 오랫동안 잠글 수 있다.

1.5 Docker Images
도커(Docker)는 널리 사용되는 가상화 기술로, 필요한 모든 리소스가 포함된 애플리케이션을 표준화된 
단위인 도커 컨테이너에 패키징할 수 있다. 이러한 방식으로 캡슐화된 소프트웨어는 추가 변경이나 설
정 과정 없이 Linux, Windows, macOS 및 대부분의 클라우드 서비스에서 실행할 수 있다. Docker 컨테
이너는 애플리케이션과 필요한 모든 도구, 프로그램 라이브러리 및 파일만 포함하기 때문에 전체 운영 
체제를 Emulate하는 기존 가상화 환경에 비해 가볍다.
예를 들어, 도커를 사용하면 데이터베이스 서버나 3D CityDB 데이터베이스 스키마를 설정할 필요없이 
몇 초 만에 3D CityDB 인스턴스를 시작하고 실행할 수 있다.
도커 이미지는 3DCityDB 소프트웨어 제품군의 다음 도구에 사용할 수 있다.
●
3D City Database
●
3D City Database Importer/Exporter
●
3D Web Map Client
●
3D CityDB Web Feature Service (WFS)
모든 이미지들은 3DCityDB DockerHub(https://hub.docker.com/u/3dcitydb)에서 사용할 수 있다.

제공된다. 보다 포괄적인 문서를 보려면 각 이미지의 개별 장을 방문하시오.

1.5.4. Web Feature Service (WFS) Docker
3D CityDB Web Feature Service (WFS) 도커 이미지는 도커화된 애플리케이션 및 워크플로우를 위한 웹 
기능 서비스의 기능을 보여준다 (WFS 설명은 7장 참조). WFS 도커를 사용하면 고급 필터 기능과 같은 
풍부한 기능 집합을 제공하는 OGC WFS 인터페이스를 통해 3D CityDB 인스턴스에 저장된 기능을 노출
할 수 있다. 기본 구성의 경우 3DCityDB(CITYDB_* variables)의 연결 자격 증명(connection credentials)
만 지정해야 한다.
모든 WFS 기능(7.4.1절)들은 이미지에서 지원된다.
도커 이미지에 대한 자세한 설명서는 Web Feature Service using Docker(7.6절)에서 사용할 수 있다.
https://3dcitydb-docs.readthedocs.io/en/latest/wfs/docker.html#wfs-docker-chapter
